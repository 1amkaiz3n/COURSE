# Exploiting server-side parameter pollution in a REST URL

Untuk menyelesaikan lab ini, masuk sebagai `administrator` dan hapus `carlos`.

## Solusi

## Pelajari behavior/perilakunya

1. Di browser Burp, picu reset password untuk user `administrator`.

2. Di **Proxy > HTTP history**, perhatikan request `POST /forgot-password` dan file JavaScript terkait `/static/js/forgotPassword.js`.

3. Klik kanan request `POST /forgot-password` lalu pilih **Send to Repeater**.

4. Di tab **Repeater**, kirim ulang request untuk memastikan responsnya konsisten.

5. Kirim berbagai request dengan modifikasi nilai parameter username untuk menentukan apakah input ditempatkan di path URL pada server-side request tanpa escaping:

   1. Kirim URL-encoded `administrator#` sebagai nilai parameter `username`.

      Perhatikan bahwa ini menghasilkan error `Invalid route`. Ini menunjukkan server mungkin memasukkan input ke dalam path request server-side, dan karakter fragment telah memotong data setelahnya. Pesan juga merujuk pada definisi API.

   2. Ubah nilai parameter `username` dari `administrator%23` menjadi URL-encoded `administrator?`, lalu kirim request.

      Perhatikan bahwa ini juga menghasilkan error `Invalid route`. Ini menunjukkan input kemungkinan ditempatkan di URL path, karena karakter `?` menandai awal query string sehingga memotong URL path.

   3. Ubah nilai parameter `username` dari `administrator%3F` menjadi `./administrator`, lalu kirim request.

      Perhatikan bahwa ini mengembalikan respons asli. Ini menunjukkan request mungkin mengakses path URL yang sama dengan request awal, yang mengindikasikan input dipakai di URL path.

   4. Ubah nilai parameter `username` dari `./administrator` menjadi `../administrator`, lalu kirim request.

      Perhatikan bahwa ini menghasilkan error `Invalid route`. Ini menunjukkan request mengakses path URL yang tidak valid.

---

## Navigasi ke definisi API

1. Ubah nilai parameter `username` dari `../administrator` menjadi `../%23`. Perhatikan respons `Invalid route`.

2. Tambahkan secara bertahap sequence `../` hingga mencapai `../../../../%23`.

   Perhatikan bahwa ini menghasilkan respons `Not found`. 
   
   ```json
   {"error": "Unexpected response from API server:\n<html>\n<head>\n    <meta charset=\"UTF-8\">\n    <title>Not Found<\/title>\n<\/head>\n<body>\n    <h1>Not found<\/h1>\n    <p>The URL that you requested was not found.<\/p>\n<\/body>\n<\/html>\n"}
    ```
   Ini menunjukkan kamu sudah keluar dari root API.

3. Pada level ini, tambahkan beberapa nama file definisi API umum ke dalam path URL. Contoh:

   ```
   username=../../../../openapi.json%23
   ```

   Perhatikan bahwa ini mengembalikan pesan error yang berisi endpoint API berikut untuk mencari user:

   ```json
   {"error": "Unexpected response from API server:\n{\n  \"openapi\": \"3.0.0\",\n  \"info\": {\n    \"title\": \"User API\",\n    \"version\": \"2.0.0\"\n  },\n  \"paths\": {\n    \"/api/internal/v1/users/{username}/field/{field}\": {\n      \"get\": {\n        \"tags\": [\n          \"users\"\n        ],\n        \"summary\": \"Find user by username\",\n        \"description\": \"API Version 1\",\n        \"parameters\": [\n          {\n            \"name\": \"username\",\n            \"in\": \"path\",\n            \"description\": \"Username\",\n            \"required\": true,\n            \"schema\": {\n        ..."}
   ```

   ```
   /api/internal/v1/users/{username}/field/{field}
   ```

   Perhatikan bahwa endpoint ini menunjukkan URL path memiliki parameter bernama `field`.

---

## Eksploitasi kerentanannya

1. Perbarui nilai parameter `username` menggunakan struktur endpoint yang telah ditemukan. Tambahkan nilai tidak valid untuk parameter `field`:

   ```
   username=administrator/field/foo%23
   ```

   Kirim request. Perhatikan bahwa ini menghasilkan error karena API hanya mendukung field `email`.

2. Tambahkan `email` sebagai nilai parameter `field`:

   ```
   username=administrator/field/email%23
   ```

   Kirim request. Perhatikan bahwa ini mengembalikan respons asli. Ini mengindikasikan aplikasi server mengenali parameter `field` yang disuntikkan dan `email` adalah field yang valid.

3. Di **Proxy > HTTP history**, periksa file JavaScript `/static/js/forgotPassword.js`. Identifikasi endpoint reset password yang merujuk pada parameter `passwordResetToken`:

   ```
   /forgot-password?passwordResetToken=${resetToken}
   ```

4. Di tab **Repeater**, ubah nilai parameter `field` dari `email` menjadi `passwordResetToken`:

   ```
   username=administrator/field/passwordResetToken%23
   ```

   Kirim request. Perhatikan bahwa ini menghasilkan error karena parameter `passwordResetToken` tidak didukung oleh versi API yang digunakan aplikasi.

5. Menggunakan endpoint `/api/` yang telah ditemukan sebelumnya, ubah versi API pada nilai parameter `username`:

   ```
   username=../../v1/users/administrator/field/passwordResetToken%23
   ```

   Kirim request. Perhatikan bahwa ini mengembalikan token reset password. Catat nilai token tersebut.

6. Di browser Burp, masukkan endpoint reset password pada address bar. Tambahkan token reset password sebagai nilai parameter `reset_token`. Contoh:

   ```
   /forgot-password?passwordResetToken=123456789
   ```

7. Atur password baru.

8. Login sebagai `administrator` menggunakan password tersebut.

9. Masuk ke **Admin panel** dan hapus `carlos` untuk menyelesaikan lab.
