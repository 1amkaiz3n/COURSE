# Exploiting server-side parameter pollution in a query string

Untuk menyelesaikan lab ini, login sebagai administrator dan hapus akun `carlos`.

## Solusi

1. Di browser Burp, picu reset password untuk user **administrator**.

2. Di **Proxy > HTTP history**, perhatikan request **POST /forgot-password** dan file JavaScript terkait **/static/js/forgotPassword.js**.

3. Klik kanan request **POST /forgot-password** lalu pilih **Send to Repeater**.

4. Di tab **Repeater**, kirim ulang request untuk memastikan responsnya konsisten.

5. Ubah nilai parameter **username** dari `administrator` menjadi username tidak valid seperti `administratorx`, lalu kirim request. Perhatikan muncul error **Invalid username**.

```json
{"type":"ClientError","code":400,"error":"Invalid username."}
```

6. Coba tambahkan pasangan parameter kedua ke request server-side menggunakan karakter **&** yang di-URL encode. Contoh:

   ```json
   username=administrator%26x=y
   ```

   Kirim request. Perhatikan error **Parameter is not supported**. Ini mengindikasikan API internal mungkin membaca `&x=y` sebagai parameter terpisah, bukan bagian dari username.

```json
{"error": "Parameter is not supported."}
```

7. Coba potong query string server-side menggunakan karakter **#** yang di-URL encode:

   ```json
   username=administrator%23
   ```

   Kirim request. Perhatikan error **Field not specified**. Ini mengindikasikan kemungkinan ada parameter tambahan bernama **field** yang terhapus oleh `#`.

```json
{"error": "Field not specified."}
```

8. Tambahkan parameter **field** dengan nilai tidak valid, lalu potong query setelahnya:

   ```json
   username=administrator%26field=x%23
   ```

   Kirim request. Perhatikan error **Invalid field**. Ini menunjukkan aplikasi server mungkin mengenali parameter **field** yang disuntikkan.

9. Lakukan brute-force nilai parameter **field**:

   * Klik kanan request **POST /forgot-password**, lalu pilih **Send to Intruder**
   * Di tab **Intruder**, tandai posisi payload pada parameter field:

     ```json
     username=administrator%26field=§x§%23
     ```
   * Di panel Payloads, pilih **Add from list**
   * Pilih daftar bawaan **Server-side variable names**, lalu mulai attack
   * Analisis hasilnya: request dengan payload **username** dan **email** mengembalikan respons **200**

10. Ubah nilai parameter **field** dari `x#` menjadi `email`:

```json
username=administrator%26field=email%23
```

Kirim request. Perhatikan bahwa ini mengembalikan respons normal. Artinya **email** adalah field yang valid.

11. Di **Proxy > HTTP history**, buka file JavaScript **/static/js/forgotPassword.js**. Perhatikan endpoint reset password yang menggunakan parameter:

```
reset_token
```

Contoh:

```
/forgot-password?reset_token=${resetToken}
```

12. Di tab **Repeater**, ubah nilai parameter **field** dari `email` menjadi `reset_token`:

```json
username=administrator%26field=reset_token%23
```

Kirim request. Perhatikan bahwa respons berisi token reset password. Catat token tersebut.

```json
{"result":"y9qk52uredvqom6t0m3oawz7xyvurvqh","type":"reset_token"}
```

13. Di browser Burp, buka endpoint reset password dan masukkan token:

```
/forgot-password?reset_token=y9qk52uredvqom6t0m3oawz7xyvurvqh
```

(gunakan token yang kamu dapatkan)

14. Set password baru.

15. Login sebagai user **administrator** menggunakan password baru tersebut.

16. Masuk ke **Admin panel**, lalu hapus user **carlos** untuk menyelesaikan lab.

Sederhananya: kamu nggak “hack sistem”, kamu cuma bikin backend-nya kebingungan sampai dia sendiri yang ngasih kunci pintunya.
