# DOM XSS

## Apa itu DOM ?

Document Object Model (DOM) adalah representasi hierarkis elemen-elemen pada halaman web yang dibuat oleh browser web. Situs web dapat menggunakan JavaScript untuk memanipulasi node dan objek DOM, serta propertinya. Manipulasi DOM itu sendiri bukanlah masalah. Bahkan, ini merupakan bagian integral dari cara kerja situs web modern. Namun, JavaScript yang menangani data secara tidak aman dapat memungkinkan berbagai serangan. Kerentanan berbasis DOM muncul ketika sebuah situs web berisi JavaScript yang mengambil nilai yang dapat dikendalikan oleh penyerang, yang dikenal sebagai sumber, dan meneruskannya ke fungsi berbahaya, yang dikenal sebagai tujuan. 


DOM XSS ini biasanya terjadi ketika javascript mengambil data dari **sumber (source)** yang di kendalikan user seperti cotohnya URL,dan meneruskannya ke **tujuan (sink)** seperti `eval()`,`innerHTML`,untuk di eksekusi.

Untuk melancarkan serangan DOM XSS,kita perlu untuk bisa mengonntrol.atau menmempatkan data ke  `source` sehingga data tersebut di masukkan ke `sink`,dan menyebabkan eksekusi javascript sembarangan.

**Sumber (Source)** paling umum untuk DOM XSS adalah **URL**, yang biasanya diakses dengan Objek `window.location`. Skenarionya,penyerang akan membuat tautan

## Taint-flow vulnerabilities

Banyak kerentanan berbasis DOM dapat ditelusuri kembali ke masalah dengan cara kode sisi klien memanipulasi data yang dapat dikendalikan oleh penyerang. 

### Apa itu taint flow?

Untuk memanfaatkan kerentanan ini, penting untuk terlebih dahulu memahami dasar-dasar taint flow antara **source** dan **sink**,atau dengan kata lain **sumber** dan **penerima**. 

**Sources**

Source adalah properti JavaScript yang menerima data yang berpotensi dikendalikan oleh penyerang.Contoh source adalah `location.search`.Properti ini penting karena membaca input dari string kueri, yang relatif mudah dikendalikan oleh penyerang. Property apapun yang dapat di kendalikan oleh penyerang sangat berpotensi, seperti URL `document.referrer`,Cookie user `document.cookie`,dll.

**Sinks**

Sink adalah fungsi JavaScript atau objek DOM yang berpotensi berbahaya yang dapat menyebabkan efek yang tidak diinginkan jika data yang dikendalikan penyerang dilewatkan ke dalamnya.Misalnya, `eval()`. Fungsi ini disebut **sink** karena memproses argumen yang diteruskan kepadanya sebagai JavaScript.Contoh sink HTML adalah `document.body.innerHTML` karena berpotensi memungkinkan penyerang untuk menyuntikkan HTML berbahaya dan mengeksekusi JavaScript sembarangan. 


Pada dasarnya, kerentanan berbasis DOM muncul ketika sebuah situs web mengirimkan data dari **sumber (sources)** ke **tujuan (sinks)**, yang kemudian menangani data tersebut dengan cara yang tidak aman dalam konteks sesi klien. 

**Sumber (Sourcs)** yang paling umum adalah **URL**, yang biasanya diakses dengan Objek `location`. Penyerang dapat membuat tautan untuk mengirim korban ke halaman yang rentan dengan muatan (payload) di bagian string kueri dan fragmen URL. Perhatikan kode berikut:

```js
goto = location.hash.slice(1)
if (goto.startsWith('https:')) {
  location = goto;
}
```

Hal ini rentan terhadap pengalihan terbuka (open redirection) berbasis DOM karena sink `location.hash`  ditangani dengan cara yang tidak aman. Jika URL berisi fragmen hash yang dimulai dengan `https:`, Kode ini mengekstrak nilai dari properti `location.hash`  dan menetapkannya sebagai properti `location`  milik `window`.

Penyerang dapat mengeksploitasi kerentanan ini dengan membuat URL berikut:

`https://www.innocent-website.com/example#https://www.evil-user.net`

Ketika korban mengunjungi URL ini, JavaScript akan mengatur nilai tersebut. properti `location`  untuk `https://www.evil-user.net` yang secara otomatis mengarahkan korban ke situs berbahaya. Perilaku ini dapat dengan mudah dieksploitasi untuk membangun serangan phishing, misalnya.

## DOM clobbering

DOM clobbering adalah teknik tingkat lanjut di mana Anda menyuntikkan HTML ke dalam halaman untuk memanipulasi DOM dan pada akhirnya mengubah perilaku JavaScript di situs web. Bentuk DOM clobbering yang paling umum menggunakan elemen anchor untuk menimpa variabel global, yang kemudian digunakan oleh aplikasi dengan cara yang tidak aman, seperti menghasilkan URL skrip dinamis. 

### Apa itu DOM clobbering? 

DOM clobbering adalah teknik di mana Anda menyuntikkan HTML ke dalam halaman untuk memanipulasi DOM dan pada akhirnya mengubah perilaku JavaScript pada halaman tersebut. DOM clobbering sangat berguna dalam kasus di mana XSS tidak memungkinkan, tetapi Anda dapat mengontrol beberapa HTML pada halaman di mana atributnya `id` atau `name` diizinkan oleh filter HTML. Bentuk paling umum dari pengubahan DOM menggunakan elemen anchor untuk menimpa variabel global, yang kemudian digunakan oleh aplikasi dengan cara yang tidak aman, seperti menghasilkan URL skrip dinamis.

Istilah "clobbering" berasal dari fakta bahwa Anda "menimpa" variabel atau properti global suatu objek dan menggantinya dengan node DOM atau koleksi HTML. Misalnya, Anda dapat menggunakan objek DOM untuk menimpa objek JavaScript lain dan mengeksploitasi nama yang tidak aman, seperti... `submit`, untuk mengganggu formulir yang sebenarnya fungsi `submit()`. 

### Cara mengeksploitasi kerentanan DOM-clobbering 

Pola umum yang digunakan oleh pengembang JavaScript adalah:

```js
var someObject = window.someObject || {};
```

Jika Anda dapat mengontrol beberapa HTML di halaman tersebut, Anda dapat menimpa `someObject` Referensi dengan node DOM, seperti anchor. Perhatikan kode berikut:

```html
<script>
    window.onload = function(){
        let someObject = window.someObject || {};
        let script = document.createElement('script');
        script.src = someObject.url;
        document.body.appendChild(script);
    };
</script>
```

Untuk mengeksploitasi kode yang rentan ini, Anda dapat menyuntikkan HTML berikut untuk menimpa `someObject` referensi dengan elemen anchor:

`<a id=someObject><a id=someObject name=url href=//malicious-website.com/evil.js>`

Karena kedua anchor menggunakan **ID yang sama**, DOM mengelompokkan keduanya menjadi satu **koleksi DOM (DOM collection)**. Kemudian vektor **DOM clobbering** menimpa referensi `someObject` dengan koleksi DOM tersebut. Atribut `name` digunakan pada elemen anchor terakhir untuk melakukan **clobber** terhadap properti `url` dari objek `someObject`, yang sebelumnya menunjuk ke script eksternal.

Teknik umum lainnya adalah menggunakan elemen **form** bersama dengan elemen seperti **input** untuk melakukan clobber terhadap properti DOM. Contohnya, melakukan clobber pada atribut `attributes` dapat memungkinkan bypass terhadap filter sisi klien (*client-side filter*) yang menggunakan properti tersebut dalam logikanya.

Meskipun filter akan melakukan enumerasi terhadap properti `attributes`, filter sebenarnya tidak akan menghapus atribut apa pun karena properti tersebut sudah di-clobber dengan sebuah node DOM. Akibatnya, kamu dapat memasukkan atribut berbahaya yang biasanya akan diblokir oleh filter.

Sebagai contoh, perhatikan injeksi berikut:

```html
<form onclick=alert(1)><input id=attributes>Click me
```

Dalam kasus ini, filter sisi klien akan melakukan traversal terhadap DOM dan menemukan elemen `form` yang masuk daftar whitelist.

Normalnya, filter akan melakukan loop terhadap properti `attributes` dari elemen `form` tersebut dan menghapus atribut apa pun yang masuk daftar blacklist.

Namun, karena properti `attributes` sudah di-clobber dengan elemen `input`, filter justru melakukan loop terhadap elemen `input` tersebut.

Karena elemen `input` memiliki nilai `length` yang tidak terdefinisi (`undefined`), kondisi untuk perulangan filter (contohnya `i < element.attributes.length`) tidak terpenuhi. Akibatnya, filter langsung berpindah ke elemen berikutnya.

Hasilnya, event `onclick` sama sekali diabaikan oleh filter, sehingga fungsi `alert()` tetap dapat dipanggil oleh browser.

Singkatnya: DOM clobbering memanfaatkan cara browser membangun objek DOM dari HTML untuk **mengubah referensi objek yang dipercaya oleh JavaScript**, lalu membuat logic keamanan membaca objek yang sudah "dirusak". Browser memang sangat rajin mengubah HTML jadi objek, lalu manusia menemukan cara membuat objek itu berperilaku aneh. Tradisi lama dalam keamanan web.

## Cara menguji serangan cross-site scripting berbasis DOM. 

Sebagian besar kerentanan **DOM XSS** dapat ditemukan dengan cepat dan terpercaya menggunakan **web vulnerability scanner milik Burp Suite**.
Untuk menguji **cross-site scripting berbasis DOM (DOM-based XSS)** secara manual, biasanya kamu perlu menggunakan browser yang memiliki **developer tools**, seperti Chrome.Kamu perlu memeriksa setiap **source** yang tersedia satu per satu, lalu menguji masing-masing source tersebut secara individual.


### Menguji HTML sinks

Untuk menguji **DOM XSS pada HTML sink**, masukkan string acak berupa karakter alfanumerik ke dalam source (misalnya `location.search`), lalu gunakan **Developer Tools** untuk memeriksa HTML dan mencari di mana string tersebut muncul.

Perlu diperhatikan bahwa opsi browser **"View source"** tidak akan bekerja untuk pengujian DOM XSS karena opsi tersebut tidak memperhitungkan perubahan pada HTML yang telah dilakukan oleh JavaScript.

Di Developer Tools Chrome, kamu dapat menggunakan **Control+F** (atau **Command+F** di macOS) untuk mencari string tersebut di dalam DOM.

Untuk setiap lokasi tempat string kamu muncul di dalam DOM, kamu perlu mengidentifikasi **konteksnya**. Berdasarkan konteks tersebut, kamu harus menyesuaikan input untuk melihat bagaimana data tersebut diproses.

Contohnya, jika string kamu muncul di dalam atribut yang menggunakan tanda kutip ganda (`"`), maka coba masukkan tanda kutip ganda ke dalam string tersebut untuk melihat apakah kamu bisa keluar (*break out*) dari konteks atribut tersebut.

Perhatikan juga bahwa browser memiliki perilaku berbeda terkait **URL encoding**.

* Chrome, Firefox, dan Safari akan melakukan **URL encode** terhadap sumber seperti `location.search` dan `location.hash`.
* IE11 dan Microsoft Edge versi sebelum Chromium **tidak** melakukan URL encode terhadap sumber tersebut.

Jika data kamu sudah di-URL encode sebelum diproses oleh aplikasi, maka serangan XSS kemungkinan besar tidak akan berhasil.

Intinya: pada DOM XSS, jangan cuma lihat source HTML awal. Yang penting adalah **DOM setelah JavaScript berjalan**, karena di situlah browser sudah mengolah data dan potensi sink bisa muncul. Browser lagi-lagi melakukan "pekerjaan rumah" sendiri, lalu security researcher harus membongkar hasil kerajinannya.


### Menguji JavaScript execution sink

Menguji **JavaScript execution sink** untuk **DOM-based XSS** sedikit lebih sulit. Pada sink jenis ini, input kamu belum tentu muncul di dalam DOM, sehingga kamu tidak bisa mencarinya secara langsung.

Sebagai gantinya, kamu perlu menggunakan **JavaScript debugger** untuk menentukan apakah dan bagaimana input kamu dikirim ke sebuah sink.

Untuk setiap source yang berpotensi, seperti `location`, pertama kamu perlu mencari bagian dalam kode JavaScript halaman yang menggunakan atau mereferensikan source tersebut.

Di **Developer Tools Chrome**, kamu dapat menggunakan:

* **Control + Shift + F**
* atau **Command + Alt + F** di macOS

untuk mencari source tersebut di seluruh kode JavaScript halaman.

Setelah menemukan lokasi di mana source tersebut dibaca, kamu dapat menggunakan **JavaScript debugger** untuk menambahkan **breakpoint** dan mengikuti bagaimana nilai dari source tersebut digunakan.

Kamu mungkin menemukan bahwa nilai dari source tersebut diberikan ke variabel lain. Jika terjadi seperti ini, kamu perlu menggunakan fungsi pencarian lagi untuk melacak variabel-variabel tersebut dan melihat apakah variabel itu akhirnya diteruskan ke sebuah sink.

Ketika kamu menemukan sebuah sink yang menerima data yang berasal dari source tersebut, kamu dapat menggunakan debugger untuk memeriksa nilainya dengan mengarahkan kursor ke variabel tersebut. Ini akan menampilkan nilai variabel sebelum dikirim ke sink.

Kemudian, sama seperti pada **HTML sink**, kamu perlu menyesuaikan input untuk melihat apakah kamu dapat menghasilkan serangan XSS yang berhasil.

Singkatnya alurnya:

**Source → Variable → Sink → Uji payload**

Karena pada JavaScript sink datanya sering "jalan-jalan" lewat beberapa variabel dulu, bukan langsung nongol di HTML seperti anak hilang yang meninggalkan jejak. Jadi debugger jadi alat utama untuk mengikuti perjalanannya.


### Menguji DOM XSS menggunakan DOM Invader

Mengidentifikasi dan mengeksploitasi **DOM XSS** pada aplikasi nyata (*in the wild*) dapat menjadi proses yang melelahkan, karena sering kali membutuhkan kamu untuk memeriksa secara manual kode JavaScript yang kompleks dan sudah diminifikasi (*minified*).

Namun, jika kamu menggunakan **browser milik Burp**, kamu dapat memanfaatkan ekstensi bawaan **DOM Invader** yang sudah tersedia. Ekstensi ini melakukan sebagian besar pekerjaan sulit untuk kamu.


## Mengeksploitasi DOM XSS dengan berbagai source dan sink

Secara prinsip, sebuah website rentan terhadap **DOM-based Cross-Site Scripting (DOM XSS)** jika terdapat jalur eksekusi (*executable path*) yang memungkinkan data berpindah dari **source** menuju **sink**.

Dalam praktiknya, setiap **source** dan **sink** memiliki karakteristik serta perilaku yang berbeda-beda. Hal ini dapat memengaruhi tingkat kemudahan eksploitasi (*exploitability*) dan menentukan teknik apa yang diperlukan.

Selain itu, script pada website mungkin melakukan validasi atau pemrosesan lain terhadap data yang harus diperhitungkan ketika mencoba mengeksploitasi sebuah kerentanan.

Ada berbagai macam **sink** yang relevan terhadap kerentanan berbasis DOM. Lihat daftar berikut untuk detailnya.



Sink `document.write` bekerja dengan elemen **script**, sehingga kamu dapat menggunakan payload sederhana seperti berikut:

```javascript
document.write('... <script>alert(document.domain)</script> ...');
```

Namun, perlu diperhatikan bahwa dalam beberapa situasi, konten yang ditulis ke `document.write` dapat memiliki konteks tambahan di sekitarnya yang harus kamu perhitungkan saat melakukan eksploitasi.

Contohnya, kamu mungkin perlu menutup elemen HTML yang sudah ada terlebih dahulu sebelum memasukkan payload JavaScript.



Sink `innerHTML` tidak menerima elemen **script** pada browser modern mana pun. Selain itu, event `svg onload` juga tidak akan dijalankan.

Artinya, kamu perlu menggunakan elemen alternatif seperti:

* `img`
* `iframe`

Event handler seperti:

* `onload`
* `onerror`

dapat digunakan bersama elemen tersebut.

Contohnya:

```javascript
element.innerHTML='... <img src=1 onerror=alert(document.domain)> ...'
```



Intinya, eksploitasi DOM XSS bergantung pada kombinasi:

**Source → Data flow → Sink → Konteks HTML/JavaScript**

Sink yang berbeda punya aturan berbeda. `document.write()` lebih langsung karena browser bisa membuat elemen baru, sedangkan `innerHTML` sudah banyak dibatasi oleh browser modern. Dunia web memang lucu: browser terus menambah pertahanan, lalu peneliti keamanan harus mencari celah logika baru dari cara browser memproses HTML dan JavaScript.



## Source dan sink pada dependency pihak ketiga

Aplikasi web modern biasanya dibangun menggunakan banyak **library dan framework pihak ketiga**, yang sering menyediakan fungsi tambahan dan kemampuan baru bagi developer.

Penting untuk diingat bahwa beberapa dari library dan framework tersebut juga dapat menjadi **source** dan **sink potensial untuk DOM XSS**.



### DOM XSS pada jQuery

Jika sebuah library JavaScript seperti **jQuery** digunakan, perhatikan sink yang dapat mengubah elemen DOM pada halaman.

Contohnya, fungsi `attr()` milik jQuery dapat mengubah atribut dari elemen DOM.

Jika data dibaca dari source yang dikontrol oleh pengguna (*user-controlled source*) seperti URL, lalu diteruskan ke fungsi `attr()`, maka ada kemungkinan nilai tersebut dapat dimanipulasi sehingga menyebabkan XSS.

Contoh:

Kode JavaScript berikut mengubah atribut `href` dari elemen anchor menggunakan data yang berasal dari URL:

```javascript
$(function() {
	$('#backLink').attr("href",(new URLSearchParams(window.location.search)).get('returnUrl'));
});
```

Kamu dapat mengeksploitasinya dengan memodifikasi URL sehingga source `location.search` berisi **JavaScript URL berbahaya**.

Setelah JavaScript halaman menerapkan URL tersebut ke atribut `href` dari link kembali (*back link*), ketika link tersebut diklik maka JavaScript akan dijalankan:

```text
?returnUrl=javascript:alert(document.domain)
```



Sink lain yang perlu diperhatikan adalah fungsi selector jQuery:

```javascript
$()
```

Fungsi ini dapat digunakan untuk memasukkan objek berbahaya ke dalam DOM.



jQuery dulu sangat populer, dan salah satu kerentanan DOM XSS klasik terjadi karena website menggunakan selector ini bersama source:

```javascript
location.hash
```

untuk animasi atau melakukan auto-scroll ke elemen tertentu pada halaman.

Perilaku ini sering diterapkan menggunakan event handler `hashchange` yang rentan, contohnya:

```javascript
$(window).on('hashchange', function() {
	var element = $(location.hash);
	element[0].scrollIntoView();
});
```

Karena nilai hash dapat dikontrol oleh pengguna, attacker dapat memanfaatkannya untuk memasukkan vektor XSS ke sink selector `$()`.

Versi jQuery yang lebih baru sudah memperbaiki kerentanan khusus ini dengan mencegah injeksi HTML ke selector ketika input diawali dengan karakter hash (`#`).

Namun, kode yang rentan masih bisa ditemukan pada aplikasi nyata.



Untuk mengeksploitasi kerentanan klasik ini, kamu perlu menemukan cara memicu event `hashchange` tanpa interaksi pengguna.

Salah satu cara paling sederhana adalah mengirim eksploit melalui iframe:

```html
<iframe src="https://vulnerable-website.com#" onload="this.src+='<img src=1 onerror=alert(1)>'">
```

Pada contoh ini:

* Atribut `src` mengarah ke halaman rentan dengan nilai hash kosong.
* Setelah iframe selesai dimuat, vektor XSS ditambahkan ke hash.
* Perubahan hash tersebut memicu event `hashchange`.


> Bahkan versi jQuery yang lebih baru pun masih rentan terhadap serangan melalui `$()`selektor sink, asalkan Anda memiliki kendali penuh atas inputnya dari source yang tidak membutuhkan prefix `#`. 



### DOM XSS pada AngularJS

Jika framework seperti **AngularJS** digunakan, ada kemungkinan JavaScript dapat dieksekusi tanpa menggunakan tanda kurung siku:

```html
<>
```

atau event handler.

Ketika sebuah website menggunakan atribut:

```html
ng-app
```

pada elemen HTML, elemen tersebut akan diproses oleh AngularJS.

Dalam kondisi ini, AngularJS akan mengeksekusi JavaScript yang berada di dalam:

```text
{{ }}
```

double curly braces.

Sintaks tersebut dapat muncul langsung di HTML maupun di dalam atribut.


Intinya: dependency pihak ketiga bukan cuma "alat bantu developer". Kadang mereka juga membawa pintu belakang logika DOM sendiri. Library menghemat waktu manusia, lalu kadang manusia harus menghabiskan waktu mencari bagaimana library itu bisa salah. Tradisi web yang sangat manusiawi.

## DOM XSS yang dikombinasikan dengan data reflected dan stored

Beberapa kerentanan yang murni berbasis DOM sepenuhnya berada dalam satu halaman. Jika sebuah script membaca data dari URL dan menulisnya ke sink yang berbahaya, maka kerentanan tersebut sepenuhnya berada di sisi klien (*client-side*).

Namun, source tidak hanya terbatas pada data yang langsung diekspos oleh browser, source juga dapat berasal dari website itu sendiri.

Sebagai contoh, website sering melakukan refleksi (*reflect*) parameter URL ke dalam respons HTML dari server. Hal ini biasanya dikaitkan dengan XSS biasa, tetapi juga dapat menyebabkan kerentanan **reflected DOM XSS**.

Pada kerentanan **reflected DOM XSS**, server memproses data dari request, lalu memasukkan kembali data tersebut ke dalam response.

Data yang direfleksikan dapat ditempatkan ke dalam:

* JavaScript string literal
* atau item data di dalam DOM, seperti field form

Kemudian sebuah script pada halaman memproses data yang direfleksikan tersebut dengan cara yang tidak aman, dan akhirnya menuliskannya ke sink yang berbahaya.

```javascript
eval('var data = "reflected string"');
```

Website juga dapat menyimpan data di server lalu menampilkannya kembali di tempat lain.

Pada kerentanan **stored DOM XSS**, server menerima data dari suatu request, menyimpannya, lalu memasukkan data tersebut ke dalam response berikutnya.

Sebuah script di dalam response berikutnya memiliki sink yang kemudian memproses data tersebut dengan cara yang tidak aman.

```javascript
element.innerHTML = comment.author
```

## Ringkasan besar:

**DOM XSS biasanya terjadi ketika:**

```
SOURCE
 ↓
(location, hash, cookie, storage, postMessage, API response)
 ↓
PROCESSING
 ↓
SINK
 ↓
(document.write, innerHTML, eval, src, href, jQuery, AngularJS)
```

Ini daftar sink yang umum ditemui. Tidak semua sink otomatis XSS, karena tetap bergantung pada **source → flow → sink → encoding/filtering**. JavaScript cuma bahasa yang diberi kekuatan besar oleh browser, lalu manusia memutuskan memakai kekuatan itu sambil berharap tidak ada lubang. Tradisi web yang abadi.
